## User story

As a coach who has spent a season giving the break-screen "How did that drill go?" prompt a
thumbs-up or thumbs-down on every drill I've run, I want those ratings to come with me when I
sign in on a new phone, when I start next season, and when the AI generates my next practice
plan, so that the only structured opinions I've ever given the app actually do something for
me instead of evaporating with my browser cache.

## Why now (four lenses)

### Product Owner
Drill thumbs-up / thumbs-down ratings are the single richest, lowest-friction signal a coach
gives the app — one tap on the break screen after a drill — and today it is stored ENTIRELY in
`localStorage` (`src/lib/drill-rating-utils.ts` writes `drill-rating:${teamId}:${drillId}`).
That means every rating is per-device and per-team: a coach who switches phones loses
everything, the rollover (0036) leaves them all behind, and the coaching signature (0037) —
which builds from `plans` rows — has no way to see them. The smallest meaningful unit of value
is one server table that persists `(coach_id, drill_id, rating, last_rated_at, run_count)` and
one route that reads/writes it through `query()`/`mutate()`, with a one-time best-effort
migration of any existing localStorage entries on the next sign-in. As the existing rating UI
already sits on the break screen and the drill picker, no new surface is added — the UI just
starts reading and writing through the network, and the picker's sort order finally survives a
new phone. Same product call, less friction with the world: ratings that compound.

### Stakeholder
This is a pure moat-deepener. The structured-coach-artifact moat is the accumulation that a
forms app cannot replicate; today drill ratings are the one piece of that accumulation we
quietly let go of every time the coach clears cookies. Moving them server-side compounds three
loops we have already built: (1) the coaching signature (0037) can extend its "recurring
drills" ranking with the coach's actual rated-up drills, not just frequency — a drill the
coach upvoted in 9 of 10 sessions is a real preference signal a generic frequency count can't
see; (2) the drill picker (0014's last-practice carryover, 0020's arc on Capture) can sort by
the coach's lifetime preference, not per-team; (3) the cross-season memory (0034) gains a
coach-level dimension distinct from its per-player dimension — "this coach has run my drill 28
times across 3 teams" is the kind of accumulation a competitor can't reproduce on day one. The
longer a coach stays, the deeper this gets — which is exactly the kind of retention-by-
accumulation the moat exists to create.

### User (the coach, between drills, breaking in a new phone after the old one died)
They sign in on a new phone. They go to the drill picker before next practice. The drills
they've upvoted across both their teams are still at the top — the picker just remembers.
Mid-break, they tap thumbs-up on the corner shooting drill; that rating is in the database
within a second, and the same drill will be sorted up tomorrow when they're on the laptop at
home looking at next week's plan. On a flaky court connection the rating is queued via the
existing `mutate()` path and written when the network returns; the local UI feedback is
instant either way. No new tap. No "log in to sync" wall. Same gesture, no leak.

### Growth
This is a retention surface with the longest half-life in the product: a coach whose
preferences travel with them is a coach who can't casually start over on a competitor. The
"show me" moment is subtle but specific: a coach signs in on a new phone in front of their
co-coach and the picker already knows their three favorite drills — "wait, it remembered?".
That is the demo that distinguishes a database app from a coaching assistant, the same kind
of moment that ships 0037 (the plan that sounds like them). Its viral angle is the screenshot
of a plan generated by the AI that opens with a drill the coach actually likes — the existing
share path for plans (already in the plans page) is the loop. And by feeding the coaching
signature (0037), it makes every future plan one degree better, which is the kind of
compounding improvement a free competitor's first plan can never have.

## Acceptance criteria

Each box maps 1:1 to a vitest or Playwright test scenario.

- [ ] A new `coach_drill_signals` table persists per-coach-per-drill rating state: `(coach_id
  uuid not null, drill_id uuid not null, rating text check (rating in ('up','down')) not null,
  run_count int not null default 0, last_rated_at timestamptz not null default now(),
  primary key (coach_id, drill_id))`. The migration uses a UNIQUE numbered prefix
  (LESSONS#0006 — no collision with any existing `0NNN_` file) and adds the table name to the
  generic data-route allow-list. NO column references any `players` row, observation text, or
  parent contact — only the coach, the drill, and the rating fact. (vitest scans the migration
  SQL for banned tokens — `player`, `dob`, `parent`, `observation` — over the DDL with `--`
  comment lines stripped first per LESSONS#0088; asserts only the four columns plus the PK.)
- [ ] `GET /api/coach-drill-signals` (authed, server-supabase) returns the caller's signals
  across all their teams: `{ signals: [{ drill_id, rating, run_count, last_rated_at }] }`,
  service-role read, scoped server-side to `eq('coach_id', user.id)`. A second coach calling
  the same route NEVER receives the first coach's signals. (vitest: two seeded coaches with
  rated drills; assert each sees only their own; assert 401 without auth.)
- [ ] `PATCH /api/coach-drill-signals` (authed) accepts `{ drill_id, rating: 'up'|'down'|null
  }` and upserts the row; `rating: null` deletes it (matches the existing `toggleDrillRating`
  semantics where tapping the same rating removes it). The route NEVER trusts a client-supplied
  `coach_id` — it always resolves it from `auth.getUser()`. (vitest: PATCH writes the row;
  re-PATCH with `null` deletes it; a PATCH with a forged `coach_id` in the body is ignored and
  the caller's own row is written instead.)
- [ ] A pure helper `mergeLocalDrillRatings(localEntries, serverSignals)` resolves the one-time
  device-handoff merge: given the LocalStorage entries the existing
  `buildTeamRatingsPrefix(teamId)` scan produces and the server signals, it returns the set of
  upserts to apply (newer `last_rated_at` wins; a server entry already present is left alone;
  a local entry with no server counterpart is queued for upsert). It writes nothing on its own
  — the route does. (vitest on the pure helper across the three cases.)
- [ ] On first sign-in after this ships, the drill picker (or wherever ratings are read) calls
  the merge once per coach and posts the result through `mutate()` / `PATCH
  /api/coach-drill-signals`; subsequent sign-ins do not re-run the merge (a `migrated_drill_ratings_at`
  marker on `coaches.preferences` records that it ran). Local-storage entries are LEFT IN PLACE
  after a successful merge (offline-safe fallback). (Playwright/component test: a seeded
  coach signed in with seeded local ratings and zero server signals ends with N server rows and
  a `preferences.migrated_drill_ratings_at` timestamp set; a second login does NOT re-issue
  the migration calls.)
- [ ] `buildCoachingSignature` (the 0037 helper) gains an optional second argument
  `drillSignals?: { drill_id, rating, run_count }[]` and, when provided, RE-RANKS its existing
  `recurring_drills` output so an upvoted drill outweighs a high-frequency-but-downvoted one;
  when `drillSignals` is undefined the function returns BYTE-IDENTICAL output to today (cold
  callers are unaffected). The plan/arc routes pass the new server signals alongside the
  existing plans fetch. (vitest: signature with no signals matches the pre-existing output
  fixture exactly; signature with up-rated drills surfaces them ahead of frequency-only
  matches; a down-rated drill is suppressed.)
- [ ] Tier / privacy: NO new `feature_*` key. The server table and routes are available to
  every tier (writing a single integer rating is not paywall-worthy; gating it would break the
  free-tier experience that the existing localStorage version already supports). The drill-
  signal payload contains ONLY `(drill_id, rating, run_count, last_rated_at)` — no team_id, no
  player references — so the signal is COACH-private and NEVER appears on any public surface
  (`/share/<token>`, `/team-card/<token>`, the sitemap from 0038, etc.). (vitest scans the
  GET payload for banned keys; Playwright on the seeded public pages asserts no drill-rating
  data renders.)
- [ ] Regression: today's localStorage `getDrillRating` / `setDrillRating` / `toggleDrillRating`
  callers continue to work — the new server-backed path is a SUPERSET, and a coach offline (or
  before the one-time merge runs) still gets the existing per-device, per-team behavior so the
  break-screen rating tap never feels broken. (vitest covers a path where the server is
  unreachable and the local helpers still return the prior `localStorage` value; the existing
  drill-rating-utils unit tests continue to pass.)

## Out of scope

- A leaderboard or any cross-coach aggregation of drill ratings ("coaches like you also rated
  this drill ↑"). The signal is strictly the coach's OWN. Cross-coach aggregation is a
  separate privacy discussion and not in this ticket.
- Exposing drill ratings on a parent or public surface. The signal is coach-private. The
  public token surfaces (0010 / 0017 / 0027 / 0026) are not touched.
- A new tier gate. The localStorage version already works for free coaches; gating the server
  version would be a hidden downgrade. The break-screen tap is part of the free experience.
- A redesign of the rating UI. v1 keeps the same break-screen and drill-picker UX; the only
  visible change is that the picker now sorts by the coach's lifetime ratings, not per-team.
  A bulk "rate drills you've run" backfill screen is a separate ticket.
- Importing or syncing drill ratings from anywhere outside the coach's own localStorage. v1
  migrates only the entries the EXISTING `drill-rating-utils.ts` wrote.
- Threading drill ratings into non-plan artifacts (parent reports, recaps, weekly star). The
  signal is for plans/arcs only — the artifacts where "what drills does this coach pick?"
  belongs.
- A separate `run_count` tracking pipeline. The migration creates the column with default 0
  and increments it from the existing drill-run-history hook (LocalStorage `drill-run-history-
  utils.ts`) as part of the same first-sign-in merge; we are NOT building a real-time
  drill-start logger in this ticket. (The increment is best-effort context; if it lands empty
  the rating still works.)

  `PATCH(request)`. Both: `createServerSupabase().auth.getUser()` → 401; then
  `createServiceSupabase()` for the read/write. The `coach_id` is ALWAYS `user.id`; a
  client-supplied `coach_id` in the body is ignored. The PATCH upserts to
  `coach_drill_signals` with `{ coach_id: user.id, drill_id, rating, last_rated_at: now() }`

- `src/lib/drill-rating-utils.ts` — keep the existing localStorage API intact for offline /

  break-screen calls IN ADDITION TO the existing local write (so the UI stays instant), or —
  cleaner — add a tiny `useCoachDrillSignals()` hook (mirror the shape of
  `src/hooks/use-tier.ts` or `src/hooks/use-active-team.ts`) that subscribes to the server
  state via `query()` and writes via `mutate()`. The break-screen / drill-picker call sites
  go through the hook so the UI is always reactive to the latest server signal.
- One-time merge — pick the right surface. A `useEffect` on the break-screen first-mount is
  fine, but `src/hooks/use-active-team.ts` or a top-level dashboard provider is cleaner: on

  `buildTeamRatingsPrefix(teamId)` for the active team, call `mergeLocalDrillRatings`, post
  the upserts through the new PATCH route in a `Promise.all`, then `mutate({ table:
  'coaches', id: user.id, set: { preferences: { ...prefs, migrated_drill_ratings_at:
